# Архитектура системы управления гостевым фондом

## Технологический стек
- **Backend**: FastAPI (Python)
- **Database**: PostgreSQL (через SQLAlchemy ORM)
- **Frontend**: HTMX + Vanilla CSS + JavaScript
- **Дополнительно**: Flatpickr (выбор дат)

## Структура базы данных

### Таблица `nomera` (Комнаты)
- `id` - уникальный идентификатор
- `nomer_komnati` - номер комнаты
- `status` - статус (Убрана/Требует уборки/На ремонте)
- `zanyat` - флаг занятости (Boolean)
- `otvetstvenniy` - ответственный сотрудник
- `kolvo_komnat`, `kolvo_krovatey` - характеристики
- Поля для совместимости: `fio_zhilca`, `telefon`, `data_zaezda`, `data_viezda` и др.

### Таблица `bookings` (Бронирования)
- `id` - уникальный идентификатор
- `nomer_id` - связь с комнатой
- `start_date`, `end_date` - даты брони (строки YYYY-MM-DD)
- `guest_name` - ФИО гостя
- `phone` - телефон
- `source` - источник брони (От кого)
- `comment` - комментарий
- `status` - статус (BOOKED/CHECKED_IN/CANCELLED/COMPLETED)
- `created_at` - дата создания

### Таблица `users` (Пользователи/Персонал)
- `id` - уникальный идентификатор
- `username`, `hashed_password` - учетные данные
- `full_name` - полное имя
- `role` - роль (admin/moderator/staff/cleaner/worker)
- `is_active` - активность
- `phone`, `telegram_id`, `comment` - контакты

## Ключевые концепции

### Разделение брони и заселения
- **Бронь (Booking)**: Запись в таблице `bookings` со статусом BOOKED
- **Заселение**: Бронь со статусом CHECKED_IN + флаг `nomer.zanyat = True`
- Одна комната может иметь несколько броней (текущую и будущие)

### Логика состояний комнаты
1. **ЗАНЯТ** - есть активная бронь на сегодня (`start_date <= today <= end_date`)
2. **РЕМОНТ** - статус "На ремонте"
3. **ГРЯЗНО** - статус "Требует уборки"
4. **БРОНЬ** - есть будущая бронь, но комната чиста
5. **ГОТОВ** - чиста, свободна, нет активной брони

### Транзитные данные
Данные из таблицы `bookings` динамически подгружаются и присваиваются объектам `Nomer` для отображения:
- В `dashboard()` - для списка комнат
- В `get_nomer_details()` - для панели деталей
- В `guest_details()` - для информации о госте
- В `update_nomer()` - для сохранения при обновлении статуса

## Основные эндпоинты

### Управление комнатами
- `GET /` - главная страница со списком комнат
- `POST /nomera` - создание комнаты
- `POST /nomera/{id}/update` - обновление статуса/персонала
- `GET /nomera/{id}/details` - детали комнаты
- `DELETE /nomera/{id}` - удаление комнаты

### Бронирование
- `GET /nomera/{id}/booking-modal` - модальное окно брони
- `POST /nomera/{id}/reserv` - создание брони
- `POST /nomera/{id}/cancel-reserv` - отмена брони
- `GET /nomera/{id}/guest-details` - детали гостя

### Календарь
- `GET /calendar` - календарь занятости
- Отображает состояние каждой комнаты на каждый день месяца
- Поддержка выбора диапазона дат для бронирования

### Персонал
- `GET /staff` - список персонала
- `POST /staff` - добавление сотрудника
- `GET /staff/{id}/details` - детали сотрудника
- `POST /staff/{id}/update` - обновление данных
- `DELETE /staff/{id}` - удаление

### Назначение персонала
- `GET /nomera/{id}/assign-staff-modal` - модальное окно назначения
- Автоматически определяет роль (cleaner/worker) по статусу комнаты

## HTMX паттерны

### Out-of-Band (OOB) Swaps
При обновлении из модального окна деталей (`source='details'`):
- Основной контент: обновленная панель деталей
- OOB swap: обновленная строка в таблице комнат
- Реализовано через `hx-swap-oob="true"` в атрибутах

### Условные target и swap
```html
hx-target="{% if source == 'details' %}closest .modal-overlay{% else %}#nomer-{{ nomer.id }}{% endif %}"
hx-swap="{% if source == 'details' %}innerHTML{% else %}outerHTML{% endif %}"
```
